01
Redirect URI Manipulation
Weak URI validation lets an attacker steal the authorization code by redirecting it to their server. This exact pattern was exploited in CVE-2014-8671 (Facebook) and countless SSO misconfigurations.
HIGHauth-server/src/routes/auth.js → validateRedirectUri()

How This Attack Works

When you click Login, the client sends a redirect_uri parameter — this tells the auth server where to send the authorization code after you log in. The auth server must check this URI against its registered allowlist.

The bug: The server uses prefix matching — it checks if your URI starts with a registered URI (minus the /callback). So http://localhost:3002/callback is registered, meaning anything starting with http://localhost:3002 passes. An attacker crafts http://localhost:3002.evil.com/steal — this passes the check and the code lands on the attacker's server.

⚠️Must be in VULNERABLE mode. In SECURE mode the server returns 400 invalid_redirect_uri before issuing any code.

02
Authorization Code Interception (PKCE Bypass)
Without PKCE, any attacker who obtains an authorization code can exchange it for access tokens. PKCE cryptographically binds the code to the originating client.
HIGHauth-server/src/routes/token.js → handleAuthCodeGrant()

How This Attack Works

PKCE adds a one-time secret to the OAuth flow. The client generates a random code_verifier and hashes it to get code_challenge = SHA256(code_verifier). Only the hash goes to the auth server. At token exchange, the client must prove it knows the original verifier.

The bug: In VULNERABLE mode, the PKCE check is completely skipped — if (VULNERABLE) { console.warn('Skipping PKCE'); }. Anyone with just the authorization code (from proxy logs, browser history, or Attack 1) can exchange it for tokens without knowing the verifier.

⚠️You need an authorization code first. Use Attack 1 to capture one, or intercept one from a normal login flow. The code is auto-filled below if one was already captured.

OAuth Consent Abuse
A malicious OAuth client requests privileged scopes (admin) that it's not registered for. In vulnerable mode, no approval gate exists — the auth server simply grants what's asked.
HIGHauth-server/src/routes/auth.js → privileged scope check

How This Attack Works

The Attacker server hosts a phishing page that initiates a real OAuth flow using the malicious-client credentials. This client requests admin scope — a privileged scope it's not registered for.

The bug: The vulnerable code has a guard that's never triggered: if (!VULNERABLE && hasPrivilegedScope && !client.trusted). Since VULNERABLE is true, the check never runs and the server grants the admin scope.

05
Refresh Token Abuse
Without refresh token rotation, a stolen refresh token gives the attacker permanent account access forever. Tokens in this lab are set to expire in 3650 days (10 years).
CRITauth-server/src/routes/token.js → handleRefreshTokenGrant()

How This Attack Works

When you log in, you get both an access_token (short-lived, ~1 hour) and a refresh_token (long-lived). The refresh token is used to get new access tokens without re-logging in.

The bug: In VULNERABLE mode, the same refresh token works indefinitely and never gets rotated. A stolen refresh token gives the attacker unlimited access as long as the account exists.

In SECURE mode: Every time a refresh token is used, the old one is revoked and a new one is issued. If an attacker uses a stolen token, the legitimate user's next refresh fails, alerting to the compromise.

⚠️You need a refresh token. Log in to the Client App first — in VULNERABLE mode the token is in localStorage. Open DevTools → Application → localStorage to find it.

06
Scope Escalation
The auth server grants whatever scopes the client requests, ignoring the registered scope list. A legitimate client registered for "read profile email" can request and receive "admin write" instead.
HIGHauth-server/src/routes/auth.js → validateScopes()

How This Attack Works

When an OAuth client registers with an auth server, it declares the scopes it needs (e.g. read profile email). The auth server should only grant the intersection of what's requested and what's registered.

The bug: In VULNERABLE mode: return requested — the server returns all requested scopes with no validation. An attacker adds admin write to the scope parameter and gets a token with those privileges.

⚠️legitimate-client is registered for: read profile email. The attack URL requests read profile email admin write — in VULNERABLE mode all are granted.

07
JWT Algorithm Confusion (alg=none)
CVE-2015-9235 — The resource server uses jwt.decode() instead of jwt.verify(). decode() only Base64-decodes the payload — it never checks the signature. A crafted token with any claims is accepted.
CRITresource-server/src/server.js → requireAuth()

How This Attack Works

A JWT has three parts: header.payload.signature. When alg=none is set in the header, no signature is required — the signature field is empty. A properly implemented server must reject alg=none tokens.

The bug: The resource server uses jwt.decode(token) which does not verify the signature. It just Base64-decodes the payload. An attacker can forge any claims (sub, scope, roles) without knowing the signing secret.

You can forge tokens entirely in your browser — the ▶ Run Forged buttons below build and fire the token automatically. No copy-pasting needed.

08
CSRF on OAuth Flow (Missing State Validation)
RFC 6749 §10.12 — The state parameter is OAuth's CSRF token. A callback endpoint that skips state validation lets an attacker trick a victim's browser into completing an OAuth flow with the attacker's authorization code, binding the victim's session to the attacker's account.
HIGHclient-app/src/server.js → /attack-callback

How This Attack Works

When a user clicks Login, the client generates a random state value and stores it server-side. After the auth server redirects back with a code, the client verifies the returned state matches what it stored. This prevents CSRF.

The bug: The /attack-callback endpoint was added for Attack 6 and intentionally skips state validation. An attacker can craft a URL that initiates an OAuth flow using their own authorization code, then trick a logged-in victim into visiting it — binding the victim's session to the attacker's account.

Authorization Server Checklist

  • Exact redirect URI matching — no wildcards, no prefix match, no substring match (RFC 6749 §3.1.2)
  • PKCE required (S256 method only) for all public clients — plain method is insufficient
  • State parameter validated on every callback — prevents CSRF on the authorization flow
  • Authorization codes expire in ≤ 5 minutes and are single-use only
  • Scope intersection enforced: granted = requested ∩ registered
  • Privileged scopes require explicit admin pre-approval per client

Token Management Checklist

  • Access tokens expire in ≤ 1 hour
  • Refresh tokens expire in ≤ 30 days
  • Refresh token rotation on every use — old token revoked immediately
  • JWT verified with jwt.verify() and algorithms: ['HS256'] — never accept alg=none
  • iss and aud claims validated on every resource server request
